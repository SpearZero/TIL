## 계층형 아키텍처의 문제는 무엇일까?

아래는 상위 수준의 관점에서 일반적인 3계층 아키텍처를 표현한 그림이다.
```
웹
↓
도메인
↓
영속성
```
- 웹 계층: 요청을 받아 도메인 혹은 비즈니스 계층에 있는 서비스로 요청을 보낸다.
- 서비스: 비즈니스 로직 수행 후 영속성 계층 호출

계층형 아키텍처의 장점과 문제점
- 장점
    - 계층을 잘 이해하고 구성한다면 웹 계층이나 영속성 계층에 독립적인 도메인 로직을 작성할 수 있다.
    - 도메인 로직에 영향을 주지 않고 웹 계층과 영속성 계층에 사용된 기술을 변경할 수 있다.
- 문제점
    - 코드에 나쁜 습관들이 스며들기 쉽게 만든다. 
    - 시간이 지날수록 소프트웨어를 점점 더 변경하기 어렵게 만드는 수많은 허점들을 노출한다.

### 계층형 아키텍처는 데이터베이스 주도 설계를 유도한다
전통적인 계층형 아키텍처의 토대는 데이터베이스다.
- 웹 계층은 도메인 계층에 의존, 도메인 계층은 영속성 계층에 의존
    - DB 중심 설계로 흐른다.

애플리케이션의 목적은 비즈니스 규칙이나 정책을 반영한 모델을 만들어서 사용자가 이를 쉽게 활용하도록 돕는 것이다.
- 이 때 상태(state)가 아닌 행동(behavior)를 중심으로 모델링한다.
- 행동이 상태를 변화시키며 비즈니스를 이끈다

보통 애플리케이션 작성시 
- 데이터베이스의 구조를 먼저 생각하고 이를 도메인 로직으로 구현했다.
- 이는 전통적인 계층형 아키텍처에서는 합리적인 방법이다. 의존성의 방향에 따라 자연스럽게 구현한 것이다.
- 비즈니스 관점에서는 맞지 않다. 먼저 도메인 로직을 만들어야 한다. 그래야 로직을 제대로 이해했는지 확인할 수 있다.
    - 비즈니스 로직 검증 이후 영속성/웹 계층을 구현한다.

ORM을 계층형 아키텍처에 적용하면 비즈니스 규칙과 영속성 로직이 섞이기 쉬워진다.

```
        ┌──────────────┐
        │   서비스      │                  [도메인 계층]
        └──────┬───────┘
               │
     ┌─────────┴─────────────┐
     ↓                       ↓
┌──────────────┐        ┌──────────────┐
│   엔티티      │<────── │  리포지토리    │  [영속성 계층]
└──────────────┘        └──────────────┘
```
위 그림과 같이, ORM에 의해 관리되는 엔티티들은 일반적으로 영속성 계층에 둔다.
- 도메인 계층에서 아래 방향으로 접근 가능하기 때문에 결국 사용된다.

결국 영속성 계층과 도메인 계층에 강한 결합이 생긴다.
- 서비스는 영속성 모델을 비즈니스 모델처럼 사용한다.
- 도메인 로직 뿐만 아니라 즉시/지연 로딩, 데이터베이스 트랜잭션, 캐시 플러시 등등 영속성 계층과 관련된 작업들을 해야 한다.

영속성과 도메인이 뒤섞이면 둘을 독립적으로 변경할 수 없고, 이는 계층형 아키텍처의 목적에 반한다.

### 지름길을 선택하기 쉬워진다
전통적인 계층형 아키텍처에서 전체적으로 적용되는 규칙
- 같은 계층에 있는 컴포넌트나 아래에 있는 계층에만 접근 가능하다.

상위 계층의 컴포넌트에 접근해야 한다면 간단하게 컴포넌트를 계층 아래로 내려버리면 된다.
- 접근 가능하게 되고 문제가 해결된다.

딱 한번은 이럴 수 있지만 갈수록 죄책감이 사라진다.
- 마감이 다가오거나 과거에 전력이 있다면 심리적 부담감은 덜해진다.

영속성 계층은 수년에 걸친 개발과 유지보수로 아래와 같이 될 확률이 높다.
```
        ┌──────────────┐
        │   서비스      │                            [도메인 계층]
        └──────┬───────┘
               │
   ┌───────────┼─────────────────────┐
   ↓           ↓                     ↓
┌──────────┐   ┌──────────────┐   ┌──────────┐
│  헬퍼    │<-> │  리포지토리   │<->│ 유틸리티  │       [영속성 계층]
└──────────┘   └──────────────┘   └──────────┘
```
영속성 계층(최하단 계층)은 컴포넌트를 아래 계층으로 내릴수록 비대해진다.
- 어떤 계층에도 속하지 않는 헬퍼, 유틸리티 컴포넌트들이 아래 계층으로 내릴 가능성이 큰 후보다.

아키텍처의 '지름길 모드'를 끄고 싶다면 계층 구조만으로는 규칙 위반을 막을 수 없기 때문에, 빌드 단계에서 강제되는 아키텍처 규칙이 필요하다.

### 테스트하기 어려워진다
일반적으로 나타내는 변화의 형태
- 계층을 건너뛰는 것이다.
- 엔티티 필드를 하나만 조작하면 되는 경우 웹 계층에서 바로 영속성 계층에 접근한다.

```

        ┌──────────────┐
        │  컨트롤러     │                   [웹 계층]
        └──────┬───────┘
               │
     ┌─────────┴───────────────┐
     │                         │ 
     │  ┌──────────────┐       │ 
     │  │   서비스      │       │           [도메인 계층]
     │  └──────┬───────┘       │  
     ↓         │               ↓ 
     ┌─────────┴───────────────┐
     ↓                         ↓
┌──────────────┐        ┌──────────────┐
│   엔티티      │<────── │  리포지토리    │   [영속성 계층]
└──────────────┘        └──────────────┘
```
처음 몇 번은 괜찮지만 자주 발생하면 두 가지 문제가 발생한다.
- 단 하나의 필드를 조작하는 것에 불과해도 도메인 로직을 웹 계층에서 구현하게 된다.
    - 단순한 작업으로 시작해도 웹 계층에 도메인 로직이 쌓이면서, 결국 핵심 로직이 애플리케이션 전반에 퍼지게 된다.
- 웹 계층 테스트에서 도메인 계층뿐만 아니라 영속성 계층도 모킹해야 한다.
    - 단위 테스트의 복잡도가 올라간다.
    - 테스트 설정이 복잡해져 테스트를 안하게 될 수 있다.

웹 컴포넌트의 규모가 커지면서 다양한 영속성 컴포넌트에 의존성이 많이 쌓이면 테스트의 복잡도를 높인다.
- 테스트 작성보다 종속성을 이해하고 목(mock)을 만드는 데 더 많은 시간이 걸리게 된다.
