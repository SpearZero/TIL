## 유스케이스 구현하기

### 생성자의 힘
SendMoneyCommand 생성자의 책임
- 클래스가 불변이기 때문에 생성자의 인자 리스트에 클래스의 각 속성에 해당하는 파라미터가 포함된다.
- 파라미터의 유효성 검증을 한다.
    - 유효하지 않은 상태의 객체를 만들기 불가능하다.

파라미터가 많다면
- 빌더 패턴을 사용할 수 있다.
    - 생성자를 private로 만들고 빌더의 build() 메서드 내부에 생성자 호출을 숨길 수 있다.

```java
new SendMoneyCommandBuilder()
    .sourceAccountId(new AccountId(41L))
    .targetAccountId(new AccountId(42L))
    // ... 다른 여러 필드를 초기화
    .build();
```
유효성 검증 로직은 생성자에 그대로 둬서 유효하지 않은 상태의 객체 생성을 막을 수 있다.

빌더 vs 생성자
- 빌더는 필드 누락을 컴파일 시점에 잡지 못해 잘못된 객체가 생성될 수 있지만, 생성자는 변경 시 컴파일 에러를 발생시켜 오류를 강제하므로 더 안전하다.
- 생성자 사용시 긴 파라미터 리스트도 충분히 깔끔하게 포매팅할 수 있고, IDE들은 파라미터명 힌트도 준다.

### 유스케이스마다 다른 입력 모델
문제
- 여러 유스케이스에서 동일한 입력 모델을 사용하고 싶어진다.
- 하지만 요구사항이 다르다.

예시
- 계좌 등록 -> 소유자 ID 필요
- 계좌 업데이트 -> 계좌 ID 필요

하나의 모델을 사용한다면
- null 허용 필요 (계좌 등록이나 업데이트시 필요한 ID가 다르다.)
- 유스케이스별로 서로 다른 유효성 검증 로직이 필요하다.
    - 유스케이스에 커스텀 검증 로직이 들어가며, 비즈니스 코드가 입력 검증 관심사로 오염된다.
- 예상하지 못한 입력 상황(예: 계좌 등록 시 계좌 ID 존재)에 대한 처리 기준이 불명확해진다.

각 유스케이스 입력 모델의 장점과 단점
- 장점
    - 유스케이스를 명확하게 한다.
    - 유스케이스와 결합을 제거해서 불필요한 부수효과를 막는다.
- 단점
    - 입력 데이터를 각 유스케이스에 맞는 모델로 매핑해야 한다.

### 비즈니스 규칙 검증하기
비즈니스 규칙 검증은 유스케이스 로직의 일부다.

입력 유효성 검증 vs 비즈니스 규칙 검증
- 입력 유효성 검증
    - 구문상의 유효성을 검증한다.
    - 선언적 방식(검증 애노테이션)으로 구현할 수 있다.
    - 도메인 상태를 필요로 하지 않는다.
- 비즈니스 규칙 검증
    - 유스케이스의 맥락에서 의미적인 검증을 수행한다.
    - 도메인 모델의 현재 상태에 접근해야 한다.

예시
- 비즈니스 규칙: "출금 계좌는 초과 출금되어서는 안 된다"
- 유효성 검증: "송금되는 금액은 0보다 커야 한다"

검증 로직을 도메인 상태 필요 여부로 구분하면, 코드 위치를 결정하고 유지보수 시 찾기 쉬워진다.
- Command/입력 모델에 있는지, 도메인 엔티티/유스케이스 로직에 있는지 구분하기 쉽다.

비즈니스 규칙 검증 구현
- 비즈니스 규칙을 도메인 엔티티 안에 넣는것이 좋다.
    - 규칙이 적용되는 로직과 가까워 이해와 유지보수가 쉽기 때문이다.
```java
package buckpal.domain;

public class Account {
    
    public boolean withdraw(Money money, AccountId targetAccountId) {
        if (!mayWithdraw(money)) {
            return false;
        }
        // ...
    }
}
```

- 어려운 경우 유스케이스에서 검증한다.
    - 도메인 엔티티를 사용하기 전에 검증해도 된다.

```java
package buckpal.application.service;

@RequiredArgsConstructor
@Transactional
public class SendMoneyService implements SendMoneyUseCase {
    // ...

    @Override
    public boolean sendMoney(SendMoneyCommand command) {
        requireAccountExists(command.getSourceAccountId());
        requireAccountExists(command.getTargetAccountId());
    }
}
```
- 유효성 검증은 실패 시 예외를 던지며, 사용자와 통신하는 어댑터가 이를 처리한다.
- 복잡한 경우에는 도메인 모델을 로드해 상태를 검증하고, 비즈니스 규칙은 도메인 엔티티에 구현한다.

### 풍부한 도메인 모델 vs 빈약한 도메인 모델

풍부한 도메인 모델(rich domain model)
- 엔티티에 가능한 많은 도메인 로직이 구현된다.
    - 상태 변경 메서드 제공 및 비즈니스 규칙에 맞는 유효한 변경만을 허용한다.
- 엔티티:
    - 상태 변경 메서드 포함 (withdraw, deposit)
    - 비즈니스 규칙 내부에서 검증
- 유스케이스:
    - 흐름만 제어 (오케스트레이션)

빈약한 도메인 모델(anemic domain model)
- 유스케이스가 모든 비즈니스 로직을 가진다.
- 엔티티:
    - getter/setter만 있음
- 유스케이스:
    - 검증
    - 상태 변경
    - 저장까지 다 수행

각자의 필요에 맞는 스타일을 자유롭게 택해서 사용하면 된다.

### 유스케이스마다 다른 출력 모델
출력은 유스케이스에 맞게 구체적이어야 하며, 불필요한 데이터를 포함하기보다 최소한만 반환하고, 추가 데이터가 필요하면 별도의 유스케이스로 제공해야 한다.
- 예: 송금하기는 boolean만 반환
    - 송금하기 맥락에서 반환할 수 있는 가장 구체적인 최소한의 값

출력 모델을 공유하면 유스케이스 간 강한 결합이 발생하고, 필드 추가 시 관련 없는 유스케이스까지 영향을 받는다. 또한 공유 모델은 점점 비대해지기 때문에 단일 책임 원칙에 따라 분리해야 한다. 같은 이유로 도메인 엔티티를 출력 모델로 사용하는 것도 변경 이유를 증가시키므로 피해야 한다.

### 읽기 전용 유스케이스는 어떨까?
읽기 전용 작업은 유스케이스로 볼 수도 있지만, 애플리케이션 코어 관점에서는 단순 데이터 조회일 수 있다.

구현 방식
- 프로젝트 맥락에서 유스케이스로 간주된다면 다른 유스케이스와 동일한 방식으로 구현한다.
- 그렇지 않다면 실제 유스케이스와 구분하기 위해 쿼리로 구현할 수 있다.
- 쿼리를 위한 인커밍 포트를 만들고, 이를 쿼리 서비스(query service)에 구현한다.

쿼리 서비스 동작
- 인커밍 포트를 구현한다.
- 아웃고잉 포트를 호출해 데이터를 로드한다.

특징
- 읽기 전용 쿼리는 쓰기 유스케이스와 코드 상에서 명확히 구분된다.
- CQS, CQRS 개념과 잘 맞는다.

주의점
- 서비스는 아웃고잉 포트를 호출하는 것 외에 다른 일을 하지 않는다.
- 여러 계층에서 같은 모델을 사용하면 지름길로 클라이언트가 아웃고잉 포트를 직접 호출할 수 있는데, 이는 피해야 한다.

```java
package buckpal.application.service;

@RequiredArgsConstructor
class GetAccountBalanceService implements GetAccountBalanceQuery {
    private final LoadAccountPort loadAccountPort;

    @Override
    public Money getAccountBalance(AccountId accountId) {
        return loadAccountPort.loadAccount(accountId, LocalDateTime.now())
            .calculateBalance();
    }
}
```

### 유지보수 가능한 소프트웨어를 만드는 데 어떻게 도움이 될까?
입출력 모델을 유스케이스별로 분리하면 원치 않는 부수효과를 방지할 수 있다.

단점
- 각 유스케이스마다 별도의 모델을 만들어야 한다.
- 엔티티와 모델 간 매핑이 필요하다.

장점
- 유스케이스를 명확하게 이해할 수 있다.
- 유지보수가 쉬워진다.
- 여러 개발자가 서로 영향을 주지 않고 동시에 작업할 수 있다.

결론
- 입력 유효성 검증과 유스케이스별 입출력 모델 분리는 지속 가능한 코드에 도움이 된다.
