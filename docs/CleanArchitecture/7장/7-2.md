## 아키텍처 요소 테스트하기

### 통합 테스트로 영속성 어댑터 테스트하기
단위 테스트보다 통합 테스트를 적용하는 것이 합리적이다.
- 어댑터 로직뿐만 아니라 데이터베이스 매핑도 검증하고 싶다.

```java
@DataJpaTest
@Import({AccountPersistenceAdapter.class, AccountMapper.class})
class AccountPersistenceAdapterTest {

    @Autowired
    private AccountPersistenceAdapter adapterUnderTest;

    @Autowired
    private ActivityRepository activityRepository;

    @Test
    @Sql("AccountPersistenceAdapterTest.sql")
    void loadsAccount() {
        Account account = adapterUnderTest.loadAccount(
            new AccountId(1L),
            LocalDateTime.of(2018, 8, 10, 0, 0));
        
        assertThat(account.getActivityWindow().getActivities()).hasSize(2);
        assertThat(account.calculateBalance()).isEqualTo(Money.of(500));
    }

    @Test
    void updatesActivities() {
        Account account = defaultAccount()
            .withBaselineBalance(Money.of(555L))
            .withActivityWindow(new ActivityWindow(
                defaultActivity()
                    .withId(null)
                    .withMoney(Money.of(1L)).build()))
            .build();
        
        adapterUnderTest.updateActivities(account);

        assertThat(activityRepository.count()).isEqualTo(1);

        ActivityJpaEntity savedActivity = activityRepository.findAll().get(0);
        assertThat(savedActivity.getAmount()).isEqualTo(1L);
    }
}
```
- @DataJpaTest는 데이터베이스 접근에 필요한 객체 네트워크를 구성한다.
- @Import를 이용해 테스트 대상 객체를 명시적으로 추가할 수 있다.

loadsAccount() 테스트
- SQL 스크립트로 DB 상태를 미리 구성한다.
- 어댑터 API로 데이터를 조회한 뒤 기대한 상태인지 검증한다.

updateActivities() 테스트
- 도메인 객체를 어댑터로 전달해 저장한다.
- Repository API로 실제 DB에 저장됐는지 검증한다.

데이터베이스를 모킹하지 않고 실제 데이터베이스를 사용한다.
- 모킹 시 커버리지는 높을 수 있지만 실제 DB와 달라 오류가 발생할 가능성이 높다.

인메모리 데이터베이스를 사용할 수도 있다.
- 설정 없이 테스트 가능해 편리하다.
- 하지만 실제 데이터베이스와 동작이 다를 수 있다.

실제 데이터베이스를 사용하면
- 테스트 환경과 운영 환경 차이를 줄일 수 있다.
- 데이터베이스별 설정이나 마이그레이션 스크립트를 따로 관리할 필요가 없다.

### 시스템 테스트로 주요 경로 테스트하기
시스템 테스트는 전체 애플리케이션을 띄우고 API를 통해 요청을 보내고, 모든 계층이 조화롭게 잘 동작하는지 검증한다.

'송금하기' 유스케이스의 시스템 테스트에서는 애플리케이션에 HTTP 요청을 보내고 계좌의 잔고를 확인하는 것을 포함해서 응답을 검증한다.

```java
@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
class SendMoneySystemTest {

    @Autowired
    private TestRestTemplate restTemplate;

    @Test
    @Sql("SendMoneySystemTest.sql")
    void sendMoney() {

        Money initialSourceBalance = sourceAccount().calculateBalance();
        Money initialTargetBalance = targetAccount().calculateBalance();

        ResponseEntity response = whenSendMoney(
            sourceAccountId(),
            targetAccountId(),
            transferredAmount());
        
        then(response.getStatusCode()).isEqualTo(HttpStatus.OK);

        then(sourceAccount().calculateBalance()).isEqualTo(initialSourceBalance.minus(transferredAmount()));

        then(targetAccount().calculateBalance())
        .isEqualTo(initialTargetBalance.plus(transferredAmount()));
    }

    private ResponseEntity whenSendMoney(
            AccountId sourceAccountId,
            AccountId targetAccountId,
            Money amount) {
        
        HttpHeaders headers = new HttpHeaders();
        headers.add("Content-Type", "application/json");
        HttpEntity<Void> request = new HttpEntity<>(null, headers);

        return restTemplate.exchange(
            "/accounts/send/{sourceAccountId}/{targetAccountId}/{amount}",
            HttpMethod.POST,
            request,
            Object.class,
            sourceAccountId.getValue(),
            targetAccountId.getValue(),
            amount.getAmount());
    }

    // 일부 헬퍼 메서드는 생략
}
```
SpringBootTest
- @SpringBootTest는 애플리케이션 전체 객체 네트워크를 띄운다.
- 랜덤 포트로 실제 애플리케이션을 실행한다.

시스템 테스트 방식
- TestRestTemplate을 이용해 실제 HTTP 요청을 보낸다.
- 응답 상태와 실제 데이터 상태를 함께 검증한다.
- MockMvc보다 프로덕션 환경에 더 가깝다.

출력 어댑터와 모킹
- 시스템 테스트에서는 실제 출력 어댑터를 사용할 수 있다.
- 하지만 서드파티 시스템은 항상 실제로 띄울 수 없기 때문에 일부는 mock으로 대체할 수 있다.
- 육각형 아키텍처에서는 출력 포트만 mock하면 되기 때문에 테스트가 쉬워진다.

테스트 DSL
- 테스트 가독성을 높이기 위해 헬퍼 메서드로 도메인 특화 언어(DSL)를 만들 수 있다.
- DSL을 사용하면 도메인 전문가도 테스트를 이해하고 피드백할 수 있다.

시스템 테스트의 역할
- 시스템 테스트는 단위/통합 테스트와 다른 종류의 버그를 발견할 수 있다.
    - 예: 계층 간 매핑 오류

시나리오 기반 테스트
- 시스템 테스트는 여러 유스케이스를 결합한 사용자 시나리오를 검증할 때 효과적이다.
- 중요한 시나리오가 통과하면 배포에 대한 신뢰를 높일 수 있다.

### 얼마만큼의 테스트가 충분할까?

라인 커버리지 같은 정량적 수치에 집착하지 말고, 얼마나 안심하고 배포할 수 있는지를 테스트의 기준으로 삼아야 한다.

프로덕션에서 발생한 버그는
- 왜 테스트가 잡지 못했는지 기록하고
- 이를 재현할 수 있는 테스트를 추가해야 한다.

육각형 아키텍처의 테스트 전략
- 도메인 엔티티 -> 단위 테스트
- 유스케이스 -> 단위 테스트
- 어댑터 -> 통합 테스트
- 주요 사용자 시나리오 -> 시스템 테스트

테스트는 기능 구현 이후의 귀찮은 작업이 아니라 개발 과정에서 함께 작성되어야 한다.

또한 리팩터링마다 쉽게 깨지는 테스트는 코드 구조에 과하게 의존하고 있는 것이므로 개선이 필요하다.


### 유지보수 가능한 소프트웨어를 만드는 데 어떻게 도움이 될까?
육각형 아키텍처는 도메인 로직과 바깥으로 향한 어댑터를 분리해 명확한 테스트 전략을 만들 수 있게 한다.
- 핵심 도메인 로직은 단위 테스트
- 어댑터는 통합 테스트

입출력 포트는 테스트에서 명확한 모킹 지점이 된다.
- 포트가 작고 역할이 명확할수록 테스트와 모킹이 쉬워진다.

테스트가 지나치게 복잡하거나 어떤 테스트를 써야 할지 모르겠다면 이는 아키텍처의 문제를 나타내는 경고 신호다.
- 테스트는 유지보수 가능한 구조로 개선하도록 돕는 역할도 한다.
