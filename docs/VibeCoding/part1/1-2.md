## 프롬프트 작성의 비법: AI와의 효과적인 소통법

프롬프트 엔지니어링(prompt engineering)
- 좋은 프롬프트를 작성하는 것
- AI의 언어를 완벽하게 구사하는 기술

명확하고 구체적인 프롬프트는 한 번에 정확한 해결책을 도출할 수 있다.

프롬프트를 구체적으로 만드는 조건
- 사용 언어 또는 환경에 대한 언급
- 결과 범위
- 요구사항 및 제약조건
- 모호한 참조 피하기
- 원하는 출력 형식

피해야 할 행동
- 소설 쓰기
- AI가 세부 사항을 제대로 채울 것이라는 가정
- 개방적인 '창의적' 프롬프트
    - 원하는 결과를 정확히 명시해야 함

프롬프트가 명확해도 단번에 완벽한 결과는 나오지 않는다.
- AI와의 반복적 상호작용이 전제된 개발 과정이 필요하다.
    - 하나의 거대한 프롬프트보다 여러 번의 점진적 요청이 더 효과적일 수 있다.
- 피드백은 구체적일수록 결과 개선에 유리하다.

프롬프트 종류
- 제로샷 프롬프트
    - 예시 없이 지시 사항만으로 작업을 요청하는 방식
- 원샷 및 퓨샷 프롬프트
    - 하나 또는 여러 개의 예시를 제공한 뒤 새로운 입력에 대한 작업을 요청
- 사고의 연쇄(CoT) 프롬프트
    - 모델에 단계별로 생각하고 최종 답변을 제공하기 전에 그 이유를 보여달라고 요청
- 역할 프롬프트
    - 특정 역할이나 정체성을 부여해 응답 방식에 영향을 주는 프롬프트
- 컨텍스트 프롬프트
    - 작업 수행에 필요한 추가 정보나 배경을 함께 제공
- 메타 프롬프트
    - 작업 내용뿐 아니라 출력 형식과 기준까지 지시하는 프롬프트
- 자기 일관성(다수 출력 및 다수결 투표)
    - 동일한 프롬프트로 여러 출력을 생성한 뒤 가장 적절한 결과를 선택
- ReAct 프롬프트
    - 추론과 행동을 결합
    - 사고할 뿐만 아니라 계산 수행, API 호출 또는 툴 사용과 같은 행동을 취한다.

프롬프트 기법을 조합해서 프롬프트를 작성할 수 있다.

모델의 한계를 이해해야 한다.
- 프롬프트가 지나치게 길거나 지시사항이 많아질 경우, 모델은 의도를 정확히 파악하지 못할 수 있다.
- 모델이 프롬프트의 일부를 무시하는 징후가 보이면, 프롬프트를 간결하게 정리하거나 여러 단계로 분리하는 것이 바람직하다.
- 잘못된 정보나 코드를 제공하는 경우 올바른 요청을 제공해야 한다.
- 복잡한 작업은 하위 작업으로 나누어 단계별로 지시하는 방식이 효과적이다.

프롬프트 안티패턴
- 모호한 프롬프트
    - 충분한 컨텍스트와 구체적인 정보를 제공해야 한다.
- 과부하 프롬프트
    - 여러 작업을 한 번에 요청하지 말고, 한 번에 한 가지 작업에 집중하도록 분할한다.
- 누락된 질문
    - 정보만 제공하고 질문이나 요청 사항이 명확하지 않은 경우
    - 무엇을 원하는지 분명하게 작성해야 한다.
- 모호한 성공 기준
    - "빠르다", "깔끔하다"와 같은 추상적인 표현 대신 "중복 코드 제거", "변수 이름 개선"처럼 구체적인 목표를 제시해야 한다.
- AI의 설명이나 출력 무시
    - AI가 확인이나 추가 설명을 요청하는 경우 이를 무시하지 않는다.
    - 대화를 전제로 하고, 오해가 발생하면 다시 설명한다.
- 일관성 부족
    - 질문 방식이나 요청 형식을 계속 바꾸면 모델이 혼란을 겪을 수 있다.
    - 하나의 프롬프트에서는 일관된 스타일을 유지한다.
- 모호한 참조
    - "위의 함수", "이전 출력"과 같은 표현은 피한다.
    - 필요한 코드나 메서드를 다시 명시하는 편이 안전하다.
