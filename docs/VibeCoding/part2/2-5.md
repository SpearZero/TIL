## 생성된 코드의 이해: 검토, 수정, 소유

개발자는 AI의 코드를 검토, 테스트, 개선한 후 코드베이스에 통합해야 한다.

### 의도에서 구현으로: AI의 해석 이해하기
AI가 생성한 코드를 명령한 프롬프트(의도)와 비교해야 한다.
- 구현이 불명확하거나 의도에 맞지 않으면 다시 확인하거나 제거한다.
- 엣지 케이스가 의도한 대로 처리되는지 확인한다.
- 프롬프트의 모호함으로 인해 AI가 해석한 지점을 찾아, 이를 수용할지 수정할지 인간이 결정한다.
- 테스트만으로는 부족할 수 있으므로, 코드를 직접 읽고 검토하는 과정은 생략하지 않는다.

### '보편적'의 문제: 많이 사용한다고 정답은 아니다
- AI는 일반적인 시나리오에 맞춘 보편적인 코드를 생성하는 경향이 있다.
- 개발자는 코드에 포함된 가정과 부족한 컨텍스트를 식별한다.
- AI가 선택한 해결 방식이 적절한지 판단하고, 필요하면 수정하거나 프롬프트를 보완한다.
- 엣지 케이스를 포함해 최종 동작을 원하는 방향으로 맞추는 책임은 개발자에게 있다.

### 코드 가독성과 구조: 패턴 및 잠재적 문제
- AI는 과도한 주석, 불명확한 변수명, 불필요하게 장황한 코드를 생성할 수 있다.
    - 프로젝트 규칙과 맞게 수정
- 지나치게 단순하거나 반대로 과도하게 복잡한 코드 구조를 점검한다.
- 오프 바이 원 오류, 예외 처리 안함, 성능 문제, 적절하지 않은 라이브러리 사용, 주석과 맞지 않는 코드, 사소한 문법오류, 구식 API 사용, 플레이스 홀더(여기에 코드 입력) 등을 주의해야 한다.

### 디버깅 전략: 오류 확인 및 수정
디버깅을 위한 여섯 단계
- 문제를 재현한다.
- 문제의 원인을 찾는다.
- 코드를 프롬프트와 대조해 확인해본다.
- AI를 활용하여 디버깅을 진행한다.
- 코드를 수정한다.
- 다시 테스트한다.

### 유지보수를 위한 리팩터링: AI 코드를 자신의 코드로 만드는 방법
개발자는 AI가 신속하게 제공한 코드를 다듬어야 한다.

리팩터링의 여섯 단계
- 코드를 스타일 가이드라인에 맞추기
- 이름과 구조 개선 (클래스명, 메서드명 등)
- 불필요한 부분 제거
- 문서 추가
- 필요시 코드 최적화
- 필요시 코드 간소화

### 테스트의 중요성: 유닛, 통합, 종단 간 테스트
AI가 생성한 코드에서 테스트가 중요한 이유
- 직접 작성하지 않은 코드가 모든 경우에 잘 작동할 것이라는 확신이 필요하다.
- 기존 기능에 영향을 주지 않는지 검증하기 위해 테스트가 필요하다.

테스트의 종류
- 단위 테스트
    - AI가 생성한 함수나 모듈 단위로 동작을 검증한다.
- 통합 테스트
    - 다른 코드와 올바른 컨텍스트에서 상호작용하는지 확인한다.
- 종단 간 테스트
    - 전체 워크플로에서 시나리오가 정상적으로 수행되는지 검증한다.

테스트는 단순히 버그를 찾는 것이 아니라 코드의 신뢰성을 보장하고 의도치 않은 변경을 방지한다.

### 요약
- 개발자가 최종 코드에 대한 책임이 있다.
- AI가 생성한 코드도 다른 코드와 동일하게 취급한다.
    - 버전 관리에 포함하고 커밋 이력을 명확히 남긴다.
- 코드의 수정이 필요하면 직접 수정하거나 AI를 통해 수정해도 된다.
- 코드를 AI, 개발자 누가 작성했는지는 중요하지 않다. 코드가 프로젝트의 요구사항 기준을 충족하는지가 중요하다.
